# 第三阶段学习记录

> 学习日期：2026-07-16
> 学习主题：RAG 切块、Offset、混合召回、索引、证据门控、Hash 与生产化优化

## 1. 第三阶段主线

第三阶段解决：

> 平台政策不依赖大模型记忆，而是从可审计知识库检索可靠原文，再让模型基于证据组织回答。

完整链路：

```text
知识导入
  -> 文本规范化
  -> 切块与 overlap
  -> 保存 Document/Chunk
  -> 生成 Embedding
  -> 写入 pgvector

用户提问
  -> 关键词召回
  -> 向量召回
  -> 候选融合与排序
  -> 可靠性门控
  -> 构造 RAG Prompt
  -> LLM 返回答案与 usedChunkIds
  -> Java 校验引用
  -> 返回答案与 Citation
```

## 2. 切块是什么

切块是把长文档拆成多个较短片段，每个片段单独保存、生成向量和检索。

```text
完整售后文档
├── chunk 0：七天无理由适用条件
├── chunk 1：不适用商品
├── chunk 2：退款审核流程
└── chunk 3：退款到账时间
```

如果整篇几千字文档只生成一个向量，退货、退款、发票和保修语义会混在一起，难以精确命中。Chunk 让检索粒度落到具体规则。

当前配置：

```text
chunkSize=500
chunkOverlap=80
```

这里按 Java `String` 字符位置计算，不是模型 Token 数。

## 3. 为什么需要 overlap

如果切点正好落在事实中间：

```text
chunk 0：定制商品、激活软件不适
chunk 1：用七天无理由退货
```

两个片段都不完整。

Overlap 让下一块向前回退：

```text
chunk 0：[0, 500)
chunk 1：[420, 900)
```

重叠区域：

```text
[420, 500) 共80个字符
```

下一块可能包含：

```text
定制商品、激活软件不适用七天无理由退货
```

代价是向量、文本和上下文存在重复，所以 overlap 不是越大越好。

## 4. 如何优先在句子边界切块

先计算硬上限：

```java
proposedEnd = Math.min(
    text.length(),
    start + chunkSize
);
```

如果还没到文档结尾，就从 `proposedEnd-1` 向前扫描，寻找：

```text
换行
。！？；
.!?
```

找到边界位置 `i` 后返回：

```java
return i + 1;
```

因为 `substring(start, end)` 不包含 end，返回 `i+1` 才会保留标点。

只扫描当前最大块的后半段：

```java
earliest = start + (proposedEnd - start) / 2;
```

原因是避免为了很早的句号产生过短 chunk。后半段找不到标点时，才使用 `proposedEnd` 硬切。

算法：

```text
最大结束位置
  -> 从后向前寻找最近句子边界
  -> 找到则在标点后结束
  -> 找不到才硬切
  -> 去除首尾空白
  -> 下一块从 end-overlap 开始
```

## 5. NLP 句法分析是什么

NLP 句法分析会识别一句话内部的语法结构：

```text
用户收到商品后，如果包装完整，可以在七天内申请退货。

主语：用户
动作：申请
对象：退货
时间：七天内
条件：收到商品、包装完整
```

它比标点切块更能处理无标点长句、法律合同和复杂条件，但会增加模型、内存、延迟和部署成本。

当前政策文档规模小、标点完整，使用：

```text
句子边界 + overlap + 离线评测
```

更符合当前目标。只有评测证明复杂长句切断导致召回下降时，才值得引入 HanLP 等句法工具或语义切块。

## 6. Offset 是什么

每个 Chunk 保存：

```text
startOffset
endOffset
```

它表示 Chunk 在 `normalizedContent` 中的字符范围：

```text
[startOffset, endOffset)
```

例如：

```text
退货规则。商品签收后七天内可申请退货。定制商品除外。
```

Chunk：

```text
商品签收后七天内可申请退货。
```

可能保存：

```text
startOffset=5
endOffset=19
```

找回文本：

```java
normalizedContent.substring(5, 19)
```

## 7. Offset 如何计算和保存

切块得到原计划 `start/end` 后，跳过首尾空白：

```java
actualStart = skipWhitespaceForward(...);
actualEnd = skipWhitespaceBackward(...);
```

真正保存：

```java
chunkText = text.substring(
    actualStart,
    actualEnd
);

new ProcessedChunk(
    index,
    actualStart,
    actualEnd,
    chunkText,
    sha256(chunkText)
);
```

因此应满足：

```java
normalizedContent.substring(
    startOffset,
    endOffset
).equals(chunkText);
```

## 8. Offset 如何进入 Citation

检索命中 Chunk 后，`SearchResponse` 带回：

```text
chunkId
documentId
chunkIndex
startOffset
endOffset
chunkText
```

LLM 只返回：

```json
{
  "usedChunkIds": [101]
}
```

Java 根据 chunk 101 从本次 hits 构造 Citation，模型不能自由填写标题、原文和 Offset。

前端可以：

```java
before = normalized.substring(0, start);
highlighted = normalized.substring(start, end);
after = normalized.substring(end);
```

高亮引用位置。

## 9. Offset 当前只能精确定位 normalizedContent

规范化会删除多余空白、统一换行，因此：

```text
original content 下标
!=
normalized content 下标
```

当前 Offset 精确指向规范化全文，不保证直接指向未经清洗的原始 `content`。

如果必须定位原始文件，需要在规范化时保存：

```text
normalizedIndex -> originalIndex
```

或者新增 `originalStartOffset/originalEndOffset`。

## 10. 关键词召回如何实现

关键词召回不调用 LLM，也不使用 Embedding。

```text
用户问题
  -> KnowledgeQueryAnalyzer
  -> 完整问题 + 领域词
  -> 查询 knowledge_chunks.chunk_text
  -> 按 chunkId 合并候选
  -> 计算 keywordScore
```

`KnowledgeQueryAnalyzer`：

1. 去除首尾空白并压缩连续空白。
2. 限制问题最多 512 字符。
3. 正则提取 Token。
4. 匹配手工电商领域词。
5. 使用 `LinkedHashSet` 去重。
6. 最多保留 10 个 term。

领域词包括：

```text
七天无理由、退货、退款、售后、换货、物流、发货、签收、订单、保修、发票、优惠、地址、支付
```

## 11. LIKE 匹配的是哪个字段

Repository：

```java
findTop10ByChunkTextContainingIgnoreCaseOrderByIdDesc(term)
```

概念 SQL：

```sql
select *
from knowledge_chunks
where lower(chunk_text)
      like lower('%退款%')
order by id desc
limit 10;
```

扫描和匹配的是：

```text
knowledge_chunks.chunk_text
```

不是完整文档、`embedding_json` 或 pgvector 向量。

关键词不能跨两个 Chunk 匹配，Overlap 可以降低完整短语正好被切断的概率。

## 12. 当前关键词召回的性能边界

一次问题可能执行：

```text
1次完整问题查询
+ 最多10个term查询
```

没有合适索引时，`LIKE '%term%'` 可能扫描大量 Chunk。`LIMIT 10` 只限制返回条数，不保证只检查 10 条。

当前约十几个 Chunk，成本很小；大规模优化方向：

```text
PostgreSQL pg_trgm + GIN
PostgreSQL 中文全文索引
Elasticsearch/OpenSearch BM25
```

## 13. 索引为什么能加速关键词匹配

索引类似书末目录：

```text
退款 -> chunk 1、chunk 7
发票 -> chunk 2、chunk 9
物流 -> chunk 3
```

倒排索引把：

```text
chunk -> 包含哪些词
```

反转成：

```text
词 -> 出现在哪些 chunk
```

查询“退款”时直接取得候选 ID，不必从所有 Chunk 开始比较。

`pg_trgm` 则把文本拆成连续三字符片段，例如：

```text
七天无理由
-> 七天无、天无理、无理由
```

GIN 保存 trigram 到行 ID 的映射，再对候选执行完整 LIKE 校验。

## 14. 当前关键词是怎么得到的

当前没有专业中文分词器，也没有让 LLM 提取关键词，而是：

```text
完整规范化问题
+ 正则 Token
+ 手工 DOMAIN_TERMS
```

例如：

```text
七天无理由退货需要什么条件
```

得到类似：

```text
七天无理由退货需要什么条件
七天无理由
七天
无理由
退货
```

局限是“钱什么时候退回来”不一定能映射到“退款”，需要同义词、中文分词、Query Rewrite 或向量召回补充。

## 15. 关键词分数怎么得到

完整问题原样出现在 Chunk 中：

```text
keywordScore=0.92
```

普通 term 命中：

```java
coverage = matchedTerms.size()
        / queryTermCount;

keywordScore = min(
    0.88,
    0.48
      + coverage * 0.32
      + min(0.08, matchedTerms.size() * 0.04)
);
```

它由基础分、关键词覆盖率和命中数量奖励组成，是项目手写的启发式分数，不是数据库自动提供的 BM25 分数。

## 16. 向量分数怎么得到

完整问题生成 Embedding：

```java
queryEmbedding = embeddingModel.embed(query)
```

向量库将问题向量与 Chunk 向量比较，返回：

```java
match.score()
```

作为 `vectorScore`。

本地内存实现明确使用余弦相似度；真实 pgvector 模式由 LangChain4j Store 完成距离计算与分数返回。

默认向量初筛：

```text
minVectorScore=0.78
```

## 17. 两路结果怎么融合

关键词和向量独立召回，按 `chunkId` 合并到同一个 `SearchCandidate`。

```text
只有关键词：finalScore=keywordScore
只有向量：finalScore=vectorScore
两者都有：matchMode=HYBRID
```

混合公式：

```java
finalScore = min(
    0.99,
    keywordScore * 0.45
      + vectorScore * 0.55
      + 0.08
);
```

不是简单相加，而是加权后增加双路命中奖励。

例如：

```text
keywordScore=0.816
vectorScore=0.86

final=0.816×0.45
     +0.86×0.55
     +0.08
     =0.9202
```

最后：

```text
finalScore >= 0.62
-> 分数降序
-> contentHash 精确去重
-> TopK
-> 上下文预算
```

LLM 不参与当前排序。

## 18. 市面成熟项目通常如何实现

当前是小型可解释基线，成熟系统通常采用：

```text
Query Understanding
  ↓
并行召回
  ├─ BM25 倒排索引 TopN
  └─ Dense Vector HNSW TopN
  ↓
RRF 融合
  ↓
Cross-Encoder Reranker
  ↓
ACL、去重、相邻块或 Parent Chunk 扩展
  ↓
上下文构建
  ↓
Grounded Answer
```

当前直接加权存在两种分数尺度不一致的问题。RRF 只使用各路线排名：

```text
RRFScore = Σ 1 / (k + rank)
```

不要求 BM25 与向量分数可直接比较。Reranker 再联合阅读问题和候选，完成精排。

## 19. 原文、Chunk、Embedding 是否真的分开存储

当前不是一行包含全部数据，而是：

```text
knowledge_documents
  -> 一篇文档一行，保存完整原文

knowledge_chunks
  -> 一个Chunk一行，一篇文档对应多行

knowledge_embeddings
  -> 一个Chunk对应一条pgvector记录
```

关系：

```text
Document 1:N Chunk
Chunk 1:1 当前向量记录
```

Chunk 表还保存 `embedding_json`，pgvector 也保存向量，因此当前是分层存储并保留冗余缓存。

## 20. 为什么保留冗余数据

```text
完整原文 -> 审计和重新切块
chunkText -> 快速检索、引用和文本快照
embeddingJson -> 启动重建时复用、管理端观察
pgvector -> 在线向量相似度搜索
```

生产化可以删除 `embedding_json`，改用独立 `knowledge_chunk_embeddings` 表保存：

```text
chunk_id
embedding_model
dimension
fingerprint
vector
status
indexed_at
```

但通常仍保留 `chunkText`，因为只存 Offset 会增加全文读取和版本一致性复杂度。

## 21. 无证据拒答如何实现

候选经过阈值、排序、去重和上下文预算后，实际进入模型的 ID 保存为：

```text
contextChunkIds
```

判断：

```java
public boolean hasReliableEvidence() {
    return !contextChunkIds.isEmpty();
}
```

`RagAnswerComposer`：

```java
if (!retrieval.hasReliableEvidence()) {
    return NO_EVIDENCE;
}
```

此时不会调用 qwen-plus。测试使用 `verifyNoInteractions(gateway)` 证明模型零调用。

还有第二层：模型阅读片段后返回 `sufficient=false`，系统返回 `MODEL_UNCERTAIN`，不勉强生成政策结论。

## 22. Hit@K 是什么

Hit@K 判断：

> 正确 Chunk 是否出现在检索返回的前 K 条中。

例如 Top5：

```text
正确Chunk排第3 -> Hit@5=1
正确Chunk排第7 -> Hit@5=0
```

多个问题：

```text
Hit@K = 命中案例数 / 总案例数
```

## 23. MRR 是什么

MRR 是平均倒数排名：

```text
第1名 -> 1/1=1.0
第2名 -> 1/2=0.5
第5名 -> 1/5=0.2
未命中 -> 0
```

```text
MRR = 所有问题 reciprocal rank 的平均值
```

Hit@K 判断有没有召回，MRR 判断正确结果排得是否靠前。

当前 `RagEvaluationService` 使用五个固定问题和 `expectedTerms` 判断相关 Chunk，只评测检索，不直接证明最终回答正确。

## 24. Hash 怎么计算

项目使用 JDK SHA-256：

```java
byte[] digest = MessageDigest
        .getInstance("SHA-256")
        .digest(value.getBytes(UTF_8));

String hash = HexFormat.of()
        .formatHex(digest);
```

过程：

```text
String
-> UTF-8字节
-> SHA-256得到256 bit/32 byte
-> 十六进制编码
-> 64字符Hash
```

相同输入、算法和编码得到相同 Hash；内容变化一个字符，Hash 通常完全变化。

```text
文档Hash=SHA-256(normalizedContent)
ChunkHash=SHA-256(chunkText)
```

## 25. 当前 Hash 用来做什么

```text
document.contentHash
-> 防止相同规范化文档重复导入

chunk.contentHash
-> 检索结果完全相同文本去重
```

当前不是通过 Hash 判断“已经向量化”。向量缓存判断主要检查：

```text
embeddingJson是否存在
是否能解析
向量维度是否匹配
pgvector记录数量
```

Hash 相同只能证明文本相同，不能证明向量由当前模型生成或索引记录存在。

## 26. 如何用 Hash 做增量向量化

生产级应生成 Embedding 指纹：

```text
fingerprint = SHA-256(
    chunkHash
    + embeddingModelName
    + embeddingModelVersion
    + embeddingDimension
    + preprocessingVersion
    + chunkStrategyVersion
)
```

只有满足：

```text
fingerprint相同
status=READY
向量记录存在
```

才复用旧向量。

## 27. 是否每次都要重新切块和向量化

不需要。

```text
先计算文档Hash
  ↓
没变化 -> 不切块、不Embedding、不写索引
  ↓
有变化 -> 重新切块
  ↓
比较新旧ChunkHash和模型指纹
  ├─ 相同 -> 复用向量
  └─ 不同 -> 只Embedding变化Chunk
```

不需要先生成新向量再比较两个高维数组。

Hash 是本地 CPU 运算；Embedding 涉及网络、模型计算、费用、限流和高维索引写入，成本差异很大。

## 28. 当前启动同步器实际行为

当前 `KnowledgeIndexSynchronizer`：

```text
启动时清空向量索引
-> 遍历所有Chunk
-> embeddingJson合法且维度一致：复用
-> 否则调用Embedding模型
-> 全量写回当前EmbeddingStore
```

因此当前每次启动会重写 pgvector，但不会必然重新调用 Embedding 模型。

这是小型学习项目的恢复策略；生产系统应使用增量任务、索引版本、`PENDING/READY/FAILED` 状态和原子版本切换。

## 29. 文本规范化如何实现

文档先执行：

1. `\r\n`、`\r` 统一成 `\n`。
2. 每行 `strip()` 删除首尾空白。
3. Tab、换页符和连续空格压缩成一个空格。
4. 合并连续空行。
5. 保留正常行之间的换行。
6. 删除全文首尾空白。

例如：

```text
原文：
  七天无理由规则  \r\n

\t普通商品   可以申请。\r\n


  定制商品除外。

规范化：
七天无理由规则
普通商品 可以申请。
定制商品除外。
```

## 30. 为什么要规范化

- 消除 Windows/Linux 换行差异。
- 避免空格、Tab 和空行数量影响 Hash。
- 让相同内容得到稳定切点和 Offset。
- 减少无意义格式占用 Embedding 输入。
- 避免生成空 Chunk。

规范化不删除标点、不改写业务词、不改变句子顺序。完整原文仍保存在 `content` 用于审计。

当前规范化适合政策文本，不适合直接处理依赖缩进的代码、复杂 Markdown 表格或 PDF 页面坐标；成熟系统应按文档类型选择解析策略。

## 31. 当前实现与生产实现的边界

当前适合：

```text
小型公共政策知识库
教学和面试项目
可解释混合召回基线
```

生产化建议：

```text
BM25倒排索引
+ pgvector HNSW
+ RRF融合
+ Cross-Encoder Reranker
+ Parent-Child Chunk
+ ACL Metadata过滤
+ 异步增量索引
+ 更大评测集
+ Citation事实一致性评测
```

推荐优化顺序：

1. `pg_trgm + GIN` 或 BM25 替换多次 LIKE 扫描。
2. 使用 RRF 替代不可直接比较的手写加权分数。
3. 召回 Top20~50 后增加 Reranker。
4. 扩大到 50~100 个标注问题并加入负样本。
5. 增加索引任务、模型版本、指纹与增量更新。
6. 增加 tenant、role、有效期和文档状态过滤。

## 32. 本阶段最关键的理解

1. 切块是检索单元，不是随意截取字符串。
2. Overlap 防止跨边界事实丢失，但会增加重复和成本。
3. Offset 指向规范化全文，模型不能自己生成引用位置。
4. 关键词召回扫描 `knowledge_chunks.chunk_text`。
5. 关键词与向量独立召回，再按 chunkId 融合。
6. 当前关键词分数和融合权重是启发式规则。
7. 无可靠证据时在调用聊天模型之前停止。
8. Hit@K 看是否召回，MRR 看正确结果排名。
9. Hash 判断内容是否变化，不单独证明向量化状态。
10. 增量索引比较内容和模型指纹，不比较新旧向量数组。

## 33. 面试表达

> 我实现了带原文追踪的 Java RAG 链路：知识导入时进行稳定文本规范化，按句子边界和 overlap 切块，并记录 chunkId、documentId 和 Offset；在线检索同时执行关键词与 pgvector 向量召回，按 chunkId 合并候选，通过阈值、融合排序、去重和上下文预算筛选证据。无可靠证据时直接阻断聊天模型调用，有证据时要求模型返回 usedChunkIds，并由 Java 校验引用必须属于本次上下文。项目还建立了 Hit@K、MRR 固定评测集。当前关键词 LIKE 和手写加权是小型基线，生产化会升级为 BM25/HNSW 并行召回、RRF 融合、Reranker 精排和带模型指纹的增量索引。

## 34. 一句话总结

> 第三阶段的重点不是“把文档转成向量”，而是建立从规范化原文、可定位 Chunk、多路召回和可靠性门控，到模型受限回答、引用校验和离线评测的完整证据链。
