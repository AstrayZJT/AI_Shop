# 第四阶段：上下文、会话记忆与结构化回答合成源码学习文档

## 1. 这一阶段解决什么问题

前三阶段分别解决了三个问题：

1. Planner：把用户自然语言转换为结构化 `AssistantPlan`。
2. Tool：根据 `AssistantAction` 找到白名单工具，并按风险策略执行或只准备。
3. RAG：从知识库检索可靠原文，并生成带 citation 的回答。

但前三阶段主要通过独立预览接口运行。正式聊天接口仍然缺少一条统一链路，也没有系统处理多轮会话的上下文预算。

第四阶段的目标是把它们串起来：

```text
POST /api/assistant/chat
  -> AssistantService
  -> AssistantAgentService
  -> AssistantContextBuilder
  -> PlannerFacade
  -> ConversationPlanResolver
  -> AssistantToolOrchestrator
  -> RagAnswerComposer（需要知识库时）
  -> AssistantAnswerComposer
  -> 保存会话摘要和本轮消息
  -> ChatResponse
```

本阶段同时删除了 `AssistantService` 中旧的关键词 `if` 路由和直接订单动作路径。现在正式聊天入口与前三阶段学习代码使用同一套 Planner、Validator、Tool Registry、RAG 和风险策略。

## 2. 先理解四种不同的数据

多轮 Agent 最容易犯的错误，是把所有文本都当成同一种“记忆”。本项目把信息分成四类。

| 数据 | 来源 | 是否可信业务事实 | 作用 |
| --- | --- | --- | --- |
| 当前消息 | 本次 HTTP 请求 | 否 | 表达当前诉求 |
| 最近消息 | `assistant_messages` | 否 | 理解代词、上下文和连续问题 |
| 会话摘要 | `assistant_sessions.summary` | 否 | 压缩较早历史，减少 Prompt 长度 |
| 订单事实 | `OrderService.listOrders(user)` | 是 | 回答订单归属、状态、金额和物流 |

关键原则是：

> 历史消息和摘要可以帮助理解用户在说什么，但不能证明用户拥有什么订单，也不能证明订单当前是什么状态。

订单事实必须在每次请求中根据当前登录用户重新查询数据库。工具真正执行前还会再次通过 `OrderService.findByOrderNo(context.user(), orderNo)` 校验归属和实时状态。

## 3. 正式入口如何进入统一 Agent 链路

入口仍然是：

```text
POST /api/assistant/chat
```

代码位于：

```text
src/main/java/com/aishop/web/AssistantController.java
```

Controller 只做两件事：

1. 通过 `AuthService.requireUser(session)` 得到当前登录用户。
2. 调用 `AssistantService.chat(...)`。

`AssistantService` 现在只负责会话层职责：

- 获取或创建 `AssistantSession`。
- 生成 `threadId`。
- 调用 `AssistantAgentService.handle(...)`。
- 保存用户消息和助手消息。
- 更新有上限的会话摘要。
- 组装原有前端需要的 `ChatResponse`。

它不再自己通过大量 `if` 判断优惠、退款、订单或商品意图，也不再直接修改订单。

## 4. AssistantContextBuilder：上下文是怎样构建的

核心代码：

```text
src/main/java/com/aishop/assistant/context/AssistantContextBuilder.java
```

入口方法：

```java
public AssistantContext build(AppUser user,
                              AssistantSession session,
                              String currentMessage,
                              String unfinishedPlanSummary)
```

### 4.1 读取最近消息

Repository 新增了分页倒序查询：

```java
List<AssistantMessage> findBySessionOrderByCreatedAtDesc(
        AssistantSession session,
        Pageable pageable);
```

Builder 只读取配置允许的最近若干条消息，不会把整个会话全部加载到 Prompt：

```java
messageRepository.findBySessionOrderByCreatedAtDesc(
        session,
        PageRequest.of(0, properties.maxRecentMessages()));
```

数据库按最新消息倒序返回，预算筛选完成后再反转为正常的时间顺序。

### 4.2 重新查询当前用户订单

```java
List<OrderResponse> userOrders = orderService.listOrders(user);
```

随后只提取回答和指代解析需要的少量字段：

```java
public record AuthoritativeOrderFact(
        String orderNo,
        String status,
        BigDecimal totalAmount,
        String shippingCarrier,
        String trackingNo) {}
```

这里没有把完整订单、地址、时间线和商品明细全部放进上下文，原因是：

- Prompt 空间有限。
- 最小必要数据更容易审计。
- 减少隐私信息暴露给模型。
- 具体工具执行时仍会重新查询完整订单。

### 4.3 上下文预算如何生效

配置位于 `application.yml`：

```yaml
assistant:
  context:
    max-characters: 8000
    max-current-message-characters: 4000
    max-summary-characters: 1200
    max-recent-messages: 6
    max-message-characters: 800
    max-order-facts: 3
```

对应配置类：

```text
src/main/java/com/aishop/config/AssistantContextProperties.java
```

Builder 使用 `Budget` 记录已经使用的字符数。默认优先级是：

```text
当前消息
-> 当前用户可信订单事实
-> 未完成计划摘要
-> 会话摘要
-> 最近消息
```

后面的历史消息预算不足时会被丢弃，不会突破 `maxCharacters`。`AssistantContext` 会返回：

```java
int estimatedCharacters;
int maxCharacters;
boolean truncated;
```

这些字段会写入日志，便于观察一次请求是否发生上下文裁剪。

### 4.4 会话摘要为什么也有上限

旧实现每轮都执行字符串追加：

```text
旧摘要 | U:用户消息 A:助手回答
```

会话越长，摘要越长，最终可能超过数据库字段长度或 Prompt 上限。

现在统一调用：

```java
contextBuilder.nextConversationSummary(current, userMessage, answer)
```

当摘要超过 `maxSummaryCharacters` 时，只保留较新的尾部，并增加“较早对话已压缩”标记。它是帮助理解的短文本，不会被当成数据库事实。

## 5. “这个订单”是如何关联上一轮的

假设第一轮是：

```text
用户：查询 ORD-12345678 的物流
助手：订单 ORD-12345678 已发货
```

第二轮是：

```text
用户：取消这个订单
```

第二轮没有订单号，如果只处理当前字符串，Planner 或本地规则只能返回 `missingSlots=[orderNo]`。

### 5.1 先从最近消息找候选订单号

`AssistantContextBuilder.resolveOrderReference(...)` 只在当前消息包含以下指代表达时工作：

```text
这个订单、该订单、那笔订单、刚才的订单、上一个订单……
```

它从最新消息向前查找 `ORD-xxxxxxxx`。

### 5.2 再和当前用户真实订单取交集

找到文本订单号后并不会直接信任。Builder 先得到：

```java
Set<String> ownedOrderNos = userOrders.stream()
        .map(OrderResponse::orderNo)
        ...
```

只有最近消息中的订单号同时存在于当前用户数据库订单集合中，才能成为 `resolvedOrderNo`。

因此下面两类信息都不能伪造可信引用：

- 用户自己在历史消息里输入的他人订单号。
- 模型生成或会话摘要中编造的订单号。

### 5.3 ConversationPlanResolver 二次约束模型计划

核心代码：

```text
src/main/java/com/aishop/assistant/context/ConversationPlanResolver.java
```

Planner 已经生成计划后，Resolver 会检查所有订单动作的 `orderNo`：

```text
QUERY_ORDER
QUERY_LOGISTICS
CANCEL_ORDER
PAY_ORDER
REQUEST_REFUND
CONFIRM_RECEIPT
UPDATE_ADDRESS
```

订单号只有两种合法来源：

1. 当前用户消息明确写出了该订单号。
2. Builder 根据“最近消息 + 当前用户真实订单”解析出的 `resolvedOrderNo`。

如果 LLM 自己编造了一个订单号，Resolver 会：

```text
删除 slots.orderNo
-> 把 orderNo 加回 missingSlots
-> 再次调用 PlanValidator.validate(...)
```

如果用户说“取消这个订单”，并且上一轮可信目标是 `ORD-12345678`，Resolver 会：

```text
slots.orderNo = ORD-12345678
missingSlots 删除 orderNo
```

这里解决的是“上下文指代和模型参数来源”问题。工具层仍然负责最后的用户归属、状态和参数校验，两层不能相互替代。

## 6. 统一应用链路 AssistantAgentService

核心代码：

```text
src/main/java/com/aishop/assistant/application/AssistantAgentService.java
```

主方法只有一条清晰链路：

```java
AssistantContext context = contextBuilder.build(user, session, message, null);
var planned = plannerFacade.plan(context.toPlannerInput());
var resolvedPlan = planResolver.resolve(context, planned);
ToolPlanExecutionResult execution = toolOrchestrator.executePlan(user, resolvedPlan);
RagAnswerResult ragAnswer = composeRagAnswer(execution);
var answer = answerComposer.compose(context, execution, ragAnswer);
```

每一步的职责分别是：

| 步骤 | 输入 | 输出 | 负责什么 |
| --- | --- | --- | --- |
| ContextBuilder | 用户、会话、消息 | `AssistantContext` | 预算、记忆、可信事实、指代 |
| PlannerFacade | `PlannerInput` | `PlannerResult` | LLM 规划、JSON 校验、本地降级 |
| PlanResolver | Context + Plan | 修正后的 Plan | 限制订单号来源、补全可信引用 |
| Orchestrator | 用户 + Plan | `TaskToolResult[]` | 排序、工具路由、风险策略 |
| RagComposer | 检索结果 | `RagAnswerResult` | 证据回答与 citation 校验 |
| AnswerComposer | Context + Tool + RAG | `AssistantComposedAnswer` | 面向用户的最终回答 |

## 7. Tool 与 RAG 为什么不重复检索

Planner 规划 `SEARCH_KNOWLEDGE` 后，`SearchKnowledgeTool` 会调用：

```java
KnowledgeRetrievalResult retrieval = knowledgeService.retrieve(query);
```

Tool 返回的数据中现在保留了同一个 `retrieval` 对象：

```java
Map.of(
    "sources", sources,
    "count", sources.size(),
    "retrieval", retrieval,
    "contextChunkIds", retrieval.contextChunkIds(),
    "contextTruncated", retrieval.contextTruncated())
```

`AssistantAgentService` 从成功的 Tool 结果中取出它，再调用：

```java
ragAnswerComposer.compose(query, retrieval)
```

这意味着一次 Agent 请求只做一次混合检索。RAG 的 Generation 阶段复用 Tool 的 Retrieval 结果，不会再向量检索一次。

`RagAnswerComposer` 仍然保留原有安全机制：

- 无可靠证据时不调用模型。
- 模型必须返回严格 JSON。
- `usedChunkIds` 必须属于本次 `contextChunkIds`。
- 最终 citation 只能来自本次召回结果。

## 8. AssistantAnswerComposer 如何生成最终回答

核心代码：

```text
src/main/java/com/aishop/assistant/answer/AssistantAnswerComposer.java
```

它不是让模型随意阅读工具日志，而是先按 `ToolExecutionStatus` 做确定性处理。

### 8.1 NEEDS_INPUT

工具缺少订单号时：

```text
继续处理前，请补充订单号。
```

此时不会调用回答模型。

### 8.2 PREPARED

高风险动作只完成准备时：

```text
已完成取消订单的前置校验……该动作需要二次确认，目前尚未执行。
```

回答层明确禁止把 `PREPARED` 描述为“取消成功”。真正的 PendingAction、确认接口和恢复执行会在下一阶段实现。

### 8.3 SUCCEEDED

只读工具成功后，根据 action 读取结构化字段：

- `QUERY_ORDER`：订单号、状态、金额、是否可取消。
- `QUERY_LOGISTICS`：状态、承运商、运单号、最新节点。
- `SEARCH_PRODUCT`：商品名和价格。
- `SEARCH_KNOWLEDGE`：交给 RAG grounded answer。

这里不解析工具返回的自然语言去猜字段，而是通过 Jackson 把结构化 `data` 转成 `JsonNode` 后读取固定路径。

### 8.4 普通聊天才调用回答模型

当计划是 `GENERAL_CHAT`，并且没有 Tool/RAG 证据可合成时，AnswerComposer 才调用：

```text
AssistantAnswerModelGateway
  -> LangChain4jAssistantAnswerModelGateway
  -> ChatModel.chat(prompt)
```

Prompt 中明确分隔：

- `authoritativeOrders` 是数据库事实。
- `conversationSummary` 和 `recentMessages` 是不可信上下文。
- 模型不得声称已完成支付、取消、退款、改地址或确认收货。

模型不可用时返回本地兜底文本，不影响 Tool 和 RAG 的确定性结果。

## 9. 会话数据保存在哪些表

本阶段没有新增状态机表，继续使用两个已有表。

### 9.1 assistant_sessions

主要字段：

```text
id
user_id
title
summary
last_intent
service_status
assigned_admin_id
last_customer_message_at
last_support_message_at
support_unread_count
customer_unread_count
```

`summary` 是有上限的会话理解文本，不是订单事实表。

### 9.2 assistant_messages

主要字段：

```text
id
session_id
role
content
created_at
```

每轮成功后分别写入一条 `user` 消息和一条 `assistant` 消息。

### 9.3 unfinishedPlanSummary 为什么现在是 null

`AssistantContext` 已经预留了 `unfinishedPlanSummary`，但本阶段还没有 `PlanRun`、`TaskRun` 和 `PendingAction` 持久化表，所以正式链路传入 `null`。

下一阶段实现状态机后，这里会保存“正在等待参数或确认的计划摘要”，服务重启后也可以恢复。

## 11. 推荐源码阅读顺序

第一次阅读建议按请求实际流向：

1. `AssistantController.chat()`
2. `AssistantService.chat()`
3. `AssistantAgentService.handle()`
4. `AssistantContextBuilder.build()`
5. `AssistantContext.toPlannerInput()`
6. `PlannerFacade.plan()`
7. `ConversationPlanResolver.resolve()`
8. `AssistantToolOrchestrator.executePlan()`
9. `SearchKnowledgeTool.execute()`

不要一开始就逐行读所有 record。先沿主链路走一遍，再回头看每个数据结构。

## 12. 面试时如何讲这一阶段

可以这样概括：

> 我把原来分散的 Planner、Tool Calling 和 RAG 预览能力接入正式客服链路，并设计了有固定预算的上下文构建器。多轮消息和会话摘要只用于语义理解，订单归属与状态每轮从数据库重新查询。针对“这个订单”一类跨轮指代，我先从历史消息找订单号，再与当前用户真实订单求交集；Planner 生成计划后还会清理没有当前消息或可信引用依据的订单号，防止模型幻觉参数进入工具层。最终回答按 ToolExecutionStatus 确定性合成，高风险动作的 PREPARED 状态明确标记为尚未执行，知识问题复用 Tool 的检索结果生成带引用的 RAG 回答。

这个亮点的重点不是“保存聊天记录”，而是：

- 有预算地选择上下文。
- 区分不可信记忆和权威业务事实。
- 约束 LLM 参数来源。
- 复用结构化 Tool/RAG 结果。
- 最终回答不夸大动作执行状态。

## 13. 当前边界和下一阶段

本阶段已经完成：

- 正式聊天统一链路。
- 有预算的短期记忆和摘要。
- 可信订单事实分层。
- “这个订单”的跨轮解析。
- Planner 订单号二次约束。
- Tool/RAG 结构化回答合成。

本阶段没有实现：

- PendingAction 数据表。
- 用户二次确认接口。
- HTTP 请求之间的计划恢复。
- `dependsOn` 条件执行和冲突分析。
- 超时、幂等和并发确认。

这些属于下一阶段“多任务编排、状态机和 PendingAction 恢复”，不能把当前的 `PREPARED` 描述成已经完成了 Human-in-the-loop 全流程。
