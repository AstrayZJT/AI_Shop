# 第五阶段：多任务编排、持久化状态机与跨请求恢复源码学习文档

## 1. 为什么第四阶段之后还需要状态机

第四阶段已经把正式客服链路统一为：

```text
Context -> Planner -> Tool -> RAG -> Answer
```

但是当时的 `AssistantToolOrchestrator` 仍然有一个前提：一次 HTTP 请求内把计划处理完。

这个前提不适用于下面两类任务：

```text
用户：取消订单
助手：请补充订单号
用户：ORD-12345678
```

以及：

```text
用户：取消 ORD-12345678
助手：该动作需要二次确认，目前尚未执行
用户：确认执行
```

中间等待用户时，服务端不能让第一个 HTTP 请求一直阻塞，也不能只把待确认信息保存在 Java 内存里。否则会出现：

- 请求线程长期占用。
- 服务重启后流程丢失。
- 多实例部署时另一台机器不知道之前执行到哪里。
- 用户下一句话再次调用 Planner，可能产生完全不同的新计划。
- 无法审计模型原计划、工具结果和确认动作。

第五阶段把 Agent 处理方式改成：

```text
执行到等待点
-> 把计划、任务、参数、结果和过期时间写入数据库
-> 当前 HTTP 请求立即返回
-> 用户下一次请求到达
-> 从数据库加载等待状态
-> 从原任务继续执行
```

这就是本项目中的“跨请求恢复”。它不是让 Java 线程一直 `wait()`，也不是把整个 Spring Service 对象序列化。

## 2. 本阶段完整调用链

正式入口仍然是：

```text
POST /api/assistant/chat
```

完整流程变为：

```text
AssistantController.chat()
-> AssistantService.chat()
-> AssistantAgentService.handle()
   -> AssistantStateMachine.findWaiting(user, session)
   -> 如果存在等待计划
      -> AssistantContextBuilder.build(..., unfinishedPlanSummary)
      -> AssistantStateMachine.resume(...)
   -> 如果不存在等待计划
      -> AssistantContextBuilder.build(...)
      -> PlannerFacade.plan(...)
      -> ConversationPlanResolver.resolve(...)
      -> AssistantStateMachine.start(...)
-> RagAnswerComposer（计划中有知识检索时）
-> AssistantAnswerComposer
-> 保存本轮会话消息和摘要
```

最重要的分支是：

```java
var waiting = stateMachine.findWaiting(user, session);
if (waiting.isPresent()) {
    workflow = stateMachine.resume(user, session, message);
} else {
    var planned = plannerFacade.plan(context.toPlannerInput());
    var resolvedPlan = planResolver.resolve(context, planned);
    workflow = stateMachine.start(user, session, resolvedPlan);
}
```

因此用户回复订单号或“确认执行”时，不会重新调用 LLM Planner。

## 3. 三张状态表分别保存什么

本阶段新增三张业务状态表。

### 3.1 assistant_plan_runs

实体：

```text
src/main/java/com/aishop/domain/AssistantPlanRun.java
```

它代表“一次完整的 Agent 计划运行”。主要字段：

| 字段 | 作用 |
| --- | --- |
| `session_id` | 计划属于哪个客服会话 |
| `user_id` | 计划属于哪个当前用户 |
| `status` | 整体运行状态 |
| `plan_json` | 通过校验的完整 `AssistantPlan` |
| `planner_source` | `LLM` 或 `RULE_FALLBACK` |
| `prompt_version` | Planner Prompt 版本 |
| `model_name` | 规划模型名称 |
| `current_task_id` | 当前等待或执行到哪个任务 |
| `expires_at` | 等待状态过期时间 |
| `completed_at` | 计划结束时间 |
| `failure_reason` | 失败原因 |
| `version` | JPA 乐观锁版本 |

查询等待计划使用复合索引：

```text
session_id, user_id, status, created_at
```

这保证系统按“当前用户 + 当前会话”查找最新等待计划，而不是只根据用户输入的 runId 查询。

### 3.2 assistant_task_runs

实体：

```text
src/main/java/com/aishop/domain/AssistantTaskRun.java
```

它代表计划中的一个任务。主要字段：

| 字段 | 作用 |
| --- | --- |
| `plan_run_id` | 所属计划 |
| `task_order` | 拓扑排序后的执行顺序 |
| `task_id` | LLM 计划中的任务 ID，例如 `t1` |
| `intent` / `action` | 意图和动作 |
| `execution_mode` | `TOOL_READ`、`ASK_CONFIRM` 等 |
| `status` | 当前任务状态 |
| `task_json` | 可恢复的完整 `AssistantTask` |
| `result_json` | 完整 `TaskToolResult` |
| `tool_name` | 实际路由到的工具 |
| `target_ref` | 订单号等目标引用 |
| `started_at` / `completed_at` | 执行时间 |
| `version` | 乐观锁版本 |

`plan_run_id + task_id` 有唯一约束，避免同一计划重复插入相同任务。

### 3.3 pending_assistant_actions

实体：

```text
src/main/java/com/aishop/domain/PendingAssistantAction.java
```

它只保存需要用户确认的高风险动作。主要字段：

| 字段 | 作用 |
| --- | --- |
| `plan_run_id` / `task_run_id` | 关联原计划和原任务 |
| `session_id` / `user_id` | 绑定当前会话和用户 |
| `action` | 当前阶段是 `CANCEL_ORDER` |
| `status` | `PENDING`、`EXECUTED`、`REJECTED`、`FAILED`、`EXPIRED` |
| `target_ref` | 订单号 |
| `arguments_json` | Tool prepare 后的规范化参数 |
| `preview_json` | 执行影响预览 |
| `expires_at` | 确认有效期 |
| `confirmed_at` / `executed_at` | 确认和执行时间 |
| `version` | 乐观锁版本 |

每个 `task_run_id` 只能对应一条 PendingAction，防止同一任务重复创建确认动作。

确认读取还使用：

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
```

同一 PendingAction 的并发确认会先竞争数据库行锁。完整的幂等键和并发压力验收属于下一阶段。

## 4. 状态枚举如何设计

计划和任务统一使用：

```java
public enum AgentRunStatus {
    PENDING,
    RUNNING,
    WAITING_INPUT,
    WAITING_CONFIRMATION,
    SUCCEEDED,
    SKIPPED,
    FAILED,
    EXPIRED
}
```

它们的含义是：

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 已持久化但还未执行 |
| `RUNNING` | 当前请求正在推进 |
| `WAITING_INPUT` | 缺少槽位，等待下一条用户消息 |
| `WAITING_CONFIRMATION` | Tool 已 prepare，等待用户确认 |
| `SUCCEEDED` | 成功结束 |
| `SKIPPED` | 条件不满足、依赖失败或用户拒绝 |
| `FAILED` | 工具、校验或确认执行失败 |
| `EXPIRED` | 等待超过有效期 |

等待状态不是线程状态，而是数据库中的业务状态。

## 5. TaskSorter：dependsOn 如何真正决定执行顺序

源码：

```text
src/main/java/com/aishop/assistant/state/TaskSorter.java
```

模型返回的 `tasks` 数组顺序不一定可靠，所以不能简单执行：

```java
for (AssistantTask task : plan.tasks())
```

`TaskSorter` 使用 Kahn 拓扑排序：

1. 统计每个任务的入度。
2. 把入度为 0 的任务放入队列。
3. 每取出一个任务，就把依赖它的任务入度减 1。
4. 新的入度为 0 的任务进入队列。
5. 最终输出拓扑顺序。

例如模型返回：

```json
[
  {"taskId":"t2","dependsOn":["t1"]},
  {"taskId":"t1","dependsOn":[]}
]
```

实际执行顺序仍然是：

```text
t1 -> t2
```

如果最终排序数量少于任务总数，说明存在循环依赖，后端拒绝执行。

## 6. TaskConflictAnalyzer：什么叫任务冲突

源码：

```text
src/main/java/com/aishop/assistant/state/TaskConflictAnalyzer.java
```

当前关注订单写动作：

```text
CANCEL_ORDER
PAY_ORDER
REQUEST_REFUND
CONFIRM_RECEIPT
UPDATE_ADDRESS
```

本阶段定义两类冲突：

1. 同一订单出现重复写动作。
2. 同一订单的 `CANCEL_ORDER` 与其他写动作组合。

例如：

```text
取消 ORD-12345678，然后支付 ORD-12345678
```

Planner 即使生成两个结构合法的任务，后端也会在调用工具前把整个 PlanRun 标记为 `FAILED`，不会猜测用户更想执行哪一个。

冲突规则在 Java 中固定，不能由模型通过修改 `reason` 或 `confidence` 绕过。

## 7. ConditionEvaluator：复杂分支怎么执行

源码：

```text
src/main/java/com/aishop/assistant/state/ConditionEvaluator.java
```

示例需求：

```text
查物流，如果还没发货就取消订单
```

Planner 生成：

```text
t1 QUERY_LOGISTICS
t2 CANCEL_ORDER
   dependsOn = [t1]
   condition = t1.ORDER_STATUS IN
               [PENDING_PAYMENT, CONFIRMED, PROCESSING]
```

条件不能写成任意 SpEL、SQL 或 JavaScript。当前只允许两个字段：

```text
ORDER_STATUS
ORDER_SHIPPED_AT
```

只允许四种操作符：

```text
EQ
IN
IS_NULL
NOT_NULL
```

`ConditionEvaluator` 从前置任务结构化 `data` 的固定路径读取值，例如：

```text
data.logistics.status
data.order.status
data.currentStatus
```

如果物流工具返回 `SHIPPED`，取消条件不满足：

```text
t1 -> SUCCEEDED
t2 -> SKIPPED
PlanRun -> SUCCEEDED
PendingAction -> 不创建
```

计划整体成功，是因为系统正确执行了分支，而不是因为每个任务都必须执行。

## 8. AssistantStateMachine.start() 做了什么

源码：

```text
src/main/java/com/aishop/assistant/state/AssistantStateMachine.java
```

`start()` 的主要步骤：

```text
1. 检查当前会话是否已有等待计划
2. 再次执行 PlanValidator
3. 创建 AssistantPlanRun
4. 拓扑排序并创建 AssistantTaskRun
5. 执行 TaskConflictAnalyzer
6. 状态改为 RUNNING
7. 调用 drive() 推进任务
```

状态机再次调用 `PlanValidator` 是纵深防御。即使未来有新入口绕过 `PlannerFacade`，非法计划也不能直接进入持久化执行层。

## 9. drive() 如何形成 Agent Loop

`drive()` 按拓扑顺序循环任务，但每个任务执行前都做四层判断：

```text
任务是否已终态
-> 依赖是否失败
-> 依赖是否完成
-> conditions 是否满足
-> missingSlots 是否为空
-> 才调用工具
```

### 9.1 缺少槽位

如果 `missingSlots` 非空：

```text
TaskRun -> WAITING_INPUT
PlanRun -> WAITING_INPUT
PlanRun.currentTaskId -> 当前任务
PlanRun.expiresAt -> 当前时间 + inputTtl
HTTP -> 立即返回追问
```

不会调用 Tool，也不会阻塞线程。

### 9.2 只读工具成功

例如订单查询：

```text
TaskRun -> RUNNING
-> AssistantToolOrchestrator.executeTask()
-> ToolExecutionStatus.SUCCEEDED
-> TaskRun -> SUCCEEDED
-> 保存 resultJson
-> 继续下一个任务
```

### 9.3 高风险工具只完成 prepare

取消订单 Tool 返回 `PREPARED` 后：

```text
创建 PendingAssistantAction
TaskRun -> WAITING_CONFIRMATION
PlanRun -> WAITING_CONFIRMATION
保存 argumentsJson、previewJson、targetRef、expiresAt
HTTP -> 返回“尚未执行，请确认”
```

`PrepareCancelOrderTool.execute()` 仍然禁止调用。真正执行发生在确认恢复路径。

## 10. 缺槽位如何跨请求恢复

源码：

```text
src/main/java/com/aishop/assistant/state/PendingInputResolver.java
```

当前支持从下一条消息中解析：

- `orderNo`
- `address`

例如等待任务是：

```json
{
  "action":"CANCEL_ORDER",
  "slots":{},
  "missingSlots":["orderNo"]
}
```

用户下一条输入：

```text
订单号是 ORD-12345678
```

恢复步骤：

```text
读取 TaskRun.taskJson
-> PendingInputResolver 填充 orderNo
-> 更新原 AssistantPlan
-> 再次 PlanValidator
-> 同步更新 PlanRun.planJson 与 TaskRun.taskJson
-> TaskRun 回到 PENDING
-> PlanRun 回到 RUNNING
-> drive() 从原计划继续
```

如果后续依赖任务也缺少同一个 `orderNo`，状态机会向直接依赖任务传播该可信输入，避免连续询问同一个订单号。

用户只输入普通文本但没有合法订单号时，任务继续保持 `WAITING_INPUT`，不会把任意文本塞进订单号字段。

## 11. 确认、拒绝和未知回答

源码：

```text
src/main/java/com/aishop/assistant/state/ConfirmationDecisionResolver.java
```

只有短而明确的确认词才会执行：

```text
确认
确认执行
确定
同意
继续执行
```

明确拒绝词：

```text
取消操作
不执行
拒绝
算了
不用了
```

下面这句话不会被误认为确认：

```text
我想确认一下取消规则
```

它返回 `UNKNOWN`，PendingAction 继续等待，回答层再次提示用户明确选择。

拒绝时：

```text
PendingAction -> REJECTED
TaskRun -> SKIPPED
PlanRun 继续推进或结束
业务订单不修改
```

## 12. 确认后为什么还要重新查订单

源码：

```text
src/main/java/com/aishop/assistant/state/ConfirmedActionExecutor.java
```

PendingAction 中保存的是 prepare 时的快照。用户可能几分钟后才确认，这段时间订单状态可能已经变化。

确认取消时不会直接相信旧的：

```text
preview.currentStatus
```

而是重新执行：

```java
var current = orderService.findByOrderNo(user, orderNo);
var cancelled = orderService.cancelOrder(
        user, current.id(), note, "AI 客服");
```

这两步重新校验：

- 当前登录用户是否拥有订单。
- 订单现在是否仍存在。
- 订单当前状态是否还允许取消。
- 实际订单 ID 是否与当前数据库一致。

如果订单已经发货，`OrderService.cancelOrder()` 会失败，PendingAction 和 TaskRun 会记录 `FAILED`，不会强行修改状态。

## 13. 为什么确认不能直接调用 Tool.execute()

`PrepareCancelOrderTool` 的职责是：

```text
验证参数
-> 查询当前用户订单
-> 检查 prepare 时状态
-> 生成影响预览
```

它的 `execute()` 故意抛出：

```java
throw new SecurityException(
    "prepare_cancel_order 是 PREPARE_ONLY 工具，禁止自动执行取消订单");
```

确认后由后端固定白名单 `ConfirmedActionExecutor` 执行实际业务 Service。

这样模型、原生 Function Calling 和普通 Orchestrator 都不能绕过确认直接调用写方法。

## 14. 过期是如何处理的

配置：

```yaml
assistant:
  state:
    input-ttl: 30m
    confirmation-ttl: 10m
```

用户下一次请求到达时，`resume()` 会检查：

```text
PlanRun.expiresAt
PendingAction.expiresAt
```

如果已经过期：

```text
PlanRun -> EXPIRED
TaskRun -> EXPIRED
PendingAction -> EXPIRED
ConfirmedActionExecutor -> 不调用
```

回答用户：

```text
等待已超时，本次计划已经过期，请重新发起请求。
```

当前采用“请求到达时惰性过期”，没有启动定时任务批量清理。惰性过期已经保证过期动作不能执行；定时归档可以在运维阶段增加。

## 15. 为什么服务重启后还能恢复

恢复不依赖以下内存对象：

- 原来的 HTTP 请求。
- `AssistantAgentService` 实例。
- LangGraph 内存 State。
- Java `CompletableFuture`。
- 本地 Map 缓存。

恢复需要的数据都在三张表里：

```text
PlanRun.planJson
TaskRun.taskJson
TaskRun.resultJson
PendingAction.argumentsJson
PendingAction.previewJson
currentTaskId
status
expiresAt
```

集成测试会：

```text
启动计划
-> 进入 WAITING_CONFIRMATION
-> EntityManager.flush()
-> EntityManager.clear()
-> 重新创建 AssistantStateMachine
-> findWaiting()
-> resume("确认执行")
```

清空 JPA 一级缓存后仍能恢复，证明状态来自数据库，而不是之前的实体对象。

## 16. 和 LangGraph4j 的关系

本阶段使用普通 Spring Service 实现状态机，没有把它包装成 LangGraph4j 图。

原因是先明确以下底层机制：

- 状态保存在哪里。
- 哪个状态允许恢复。
- 用户确认如何重新校验业务数据。
- 事务何时提交。
- 条件和依赖如何求值。

项目中的 LangGraph4j 目前仍然只是轻量 checkpoint 调用，不能在简历中声称“使用 LangGraph4j 完成复杂状态编排”。

后续可以把 `start -> drive -> wait -> resume` 映射为图节点，但数据库中的 PlanRun、TaskRun 和 PendingAction 仍然需要保留，因为它们是业务审计与权限边界，不应只存在于图框架 checkpoint 中。

## 19. 面试时如何讲这一阶段

可以这样描述：

> 我为电商 AI 客服实现了持久化多任务状态机。LLM 只负责生成带 dependsOn 和 conditions 的任务计划，Java 后端使用拓扑排序、冲突分析和白名单条件求值决定实际执行顺序。针对缺订单号和取消订单确认，HTTP 请求不会阻塞等待，而是把 PlanRun、TaskRun 和 PendingAction 持久化为 WAITING_INPUT 或 WAITING_CONFIRMATION；用户下一轮输入到达后根据当前登录用户和会话恢复原计划，不重新调用 Planner。确认取消时使用数据库行锁读取 PendingAction，并重新校验订单归属和实时状态后才调用 OrderService，过期或拒绝都不会修改订单。

这里真正值得讲的不是“用了几个状态枚举”，而是：

- LLM 计划和 Java 执行权限分离。
- 等待用户不占用 HTTP 线程。
- 中断点持久化，服务重启可以恢复。
- conditions 只允许固定字段和运算符。
- prepare 和 execute 分离。
- 确认时重新校验业务事实。

## 20. 当前实现边界

本阶段已经实现：

- `TaskSorter` 拓扑排序。
- `TaskConflictAnalyzer`。
- `ConditionEvaluator`。
- `AssistantStateMachine.start()/resume()`。
- `AssistantPlanRun`、`AssistantTaskRun`、`PendingAssistantAction`。
- 缺订单号后的下一轮恢复。
- 取消订单 prepare、确认、拒绝和过期。
- 服务重启后的数据库恢复。
- 正式聊天入口接入状态机。

仍属于下一阶段的内容：

- 独立的 PendingAction 确认 REST API。
- `clientRequestId` 幂等键。
- 重复点击与并发确认压力测试。
- 更完整的跨用户攻击测试。
- Planner、Tool、确认人和失败原因的统一审计查询接口。
- 支付、退款、改地址等其他高风险动作的确认执行器。
- 定时清理长期过期状态。

因此当前可以准确地说“实现了取消订单的持久化 Human-in-the-loop 状态恢复”，不能说“所有电商写操作都已经完成统一安全确认”。
