# 第二阶段学习记录

> 学习日期：2026-07-15
> 学习主题：Tool Calling、工具注册表、安全执行、自定义 Tool 与 `@Tool` 的取舍

## 1. 第二阶段到底完成了什么

第一阶段负责：

```text
用户自然语言
  -> LLM Planner
  -> AssistantPlan
  -> Java 校验后的 PlannerResult
```

第二阶段负责：

```text
PlannerResult
  -> 根据 action 找到 Java Tool
  -> 校验工具参数和当前用户
  -> 判断工具风险
  -> 只读工具执行
  -> 高风险工具只准备
  -> 返回结构化任务结果
```

第二阶段不是简单地“让 LLM 调用一个 Java 方法”，而是建立了模型与业务系统之间的确定性执行边界。

核心组件：

| 组件 | 职责 |
|---|---|
| `AssistantTool` | 统一工具接口 |
| `AssistantToolRegistry` | 工具白名单与路由表 |
| `AssistantToolOrchestrator` | 任务排序、状态判断和工具执行 |
| `ToolPolicy` | 工具风险和自动执行权限 |
| `ToolContext` | 绑定当前登录用户和请求标识 |
| `ToolArguments` | 校验和规范化模型参数 |
| `PreparedToolCall` | 保存经过后端认可的调用参数和预览 |
| `TaskToolResult` | 返回统一执行状态 |

## 2. 第二阶段包含两条路线

当前项目保留了两条并列路线，而不是把它们前后串联。

### 2.1 Planner 驱动主线

```text
用户输入
  -> 第一阶段 Planner
  -> AssistantPlan
  -> Orchestrator
  -> Registry 按 AssistantAction 查找
  -> Tool
```

这条路线适合：

- 多任务拆分。
- `dependsOn` 依赖关系。
- `conditions` 条件描述。
- 咨询与执行意图边界。
- 可审计的完整任务计划。

### 2.2 原生 Function Calling 实验路线

```text
用户输入 + ToolSpecification
  -> LLM
  -> ToolExecutionRequest
  -> Orchestrator
  -> Registry 按工具名查找
  -> Tool
```

这条路线适合：

- 简单、直接的工具选择。
- 学习 LangChain4j 原生 Tool Calling 协议。
- 验证 `ToolSpecification`、`ToolChoice.AUTO` 和 `ToolExecutionRequest`。

两条路线共用同一套后端安全层：

```text
AssistantToolRegistry
AssistantToolOrchestrator
ToolPolicy
ToolContext
ToolArguments
具体 AssistantTool
```

## 3. 原生 Function Calling 不是 Planner 的下一步

如果 Planner 已经生成：

```json
{
  "action": "QUERY_ORDER",
  "slots": {
    "orderNo": "ORD-12345678"
  }
}
```

后端已经可以确定：

```text
QUERY_ORDER -> QueryOrderTool
```

此时不需要再把所有 `ToolSpecification` 发给 LLM，让模型重新选择一次工具。

正确主链：

```text
Planner 决定 QUERY_ORDER
  -> Registry 确定性找到 QueryOrderTool
  -> 进入安全执行
```

不推荐：

```text
Planner 决定 QUERY_ORDER
  -> 再调用一次 LLM 选择工具
  -> 增加成本、延迟和不一致风险
```

因此，第二阶段的必要部分是“Plan 到安全工具执行”；原生 Function Calling 是替代入口和学习支线。

## 4. LLM 怎么知道 Planner 应该填写哪个 Action

Planner 路线没有读取 `ToolSpecification`。LLM 是通过 `PlannerPromptFactory` 的 System Prompt 得知可用 Action 的。

Prompt 中列出了：

```text
QUERY_ORDER
QUERY_LOGISTICS
SEARCH_PRODUCT
SEARCH_KNOWLEDGE
CANCEL_ORDER
PAY_ORDER
REQUEST_REFUND
...
```

同时给出动作契约：

```text
QUERY_ORDER:
intent=ORDER,
mode=TOOL_READ,
slots=[orderNo],
required=[orderNo]

CANCEL_ORDER:
intent=ORDER,
mode=ASK_CONFIRM,
slots=[orderNo,note],
required=[orderNo]
```

因此 Planner 的工作可以理解为受约束分类：

```text
用户自然语言
  -> LLM 理解语义
  -> 从 Prompt 给出的 Action 集合中选择标签
  -> 提取 slots
  -> 输出 AssistantPlan JSON
```

LLM 不会自动读取 Java 枚举、Spring Bean 或 Tool Registry。Prompt 是它获得 Action 信息的来源，Java Validator 是最终契约来源。

## 5. Action 与 ExecutionMode 的组合从哪里来

组合关系不是 LLM 推导的，而是开发者根据业务风险手动定义的。

```text
QUERY_ORDER       -> TOOL_READ
QUERY_LOGISTICS   -> TOOL_READ
SEARCH_PRODUCT    -> TOOL_READ
CANCEL_ORDER      -> ASK_CONFIRM
PAY_ORDER         -> ASK_CONFIRM
REQUEST_REFUND    -> ASK_CONFIRM
GENERAL_CHAT      -> ANSWER_ONLY
```

后端在 `PlanValidator.expectedModes()` 中构建：

```java
result.put(
        AssistantAction.CANCEL_ORDER,
        ExecutionMode.ASK_CONFIRM);
```

当模型返回：

```text
action=CANCEL_ORDER
executionMode=TOOL_READ
```

校验过程是：

```java
ExecutionMode expectedMode =
        EXPECTED_MODES.get(task.action());
```

代入后：

```text
expectedMode        = ASK_CONFIRM
task.executionMode  = TOOL_READ

ASK_CONFIRM != TOOL_READ
  -> true
  -> 校验失败
```

Jackson 只检查两个枚举值分别是否合法，不知道它们组合后的业务含义，所以还需要 `PlanValidator`。

## 6. ToolPolicy 中的风险表达式

构造器规则：

```java
if (riskLevel != ToolRiskLevel.READ_ONLY
        && autoExecutable) {
    throw new IllegalArgumentException(
            "只有 READ_ONLY 工具可以自动执行");
}
```

中文含义：

```text
如果工具不是只读工具
并且还被配置为允许自动执行
那么这是危险配置，立即拒绝
```

真值表：

| riskLevel | autoExecutable | 结果 |
|---|---:|---|
| `READ_ONLY` | `true` | 允许 |
| `READ_ONLY` | `false` | 允许，属于更保守配置 |
| `PREPARE_ONLY` | `false` | 允许 |
| `PREPARE_ONLY` | `true` | 抛出异常 |

这里保护的是 Java 工具配置，不是直接校验 LLM 计划。

两套安全检查的分工：

```text
PlanValidator
  -> 防止 LLM 把 CANCEL_ORDER 写成 TOOL_READ

ToolPolicy
  -> 防止开发者把非只读 Tool 配成自动执行
```

## 7. AssistantToolRegistry 是什么

`AssistantToolRegistry` 是工具白名单和路由表。

内部维护两张 Map：

```java
Map<AssistantAction, AssistantTool> byAction;
Map<String, AssistantTool> byName;
```

内容类似：

```text
QUERY_ORDER       -> QueryOrderTool
CANCEL_ORDER      -> PrepareCancelOrderTool

"query_order"          -> QueryOrderTool
"prepare_cancel_order" -> PrepareCancelOrderTool
```

两张表分别服务：

```text
Planner 路线
  -> find(AssistantAction)

原生 Function Calling
  -> find(String toolName)
```

模型即使编造：

```text
delete_database
```

Registry 中不存在该名称，只会返回 `Optional.empty()`，Orchestrator 再转换为 `NOT_SUPPORTED`，不会反射调用任意 Java 方法。

## 8. Registry 为什么启动时能发现所有工具

这里存在两层注册。

### 第一层：Spring 注册 Tool Bean

具体工具满足：

```java
@Component
public class QueryOrderTool
        implements AssistantTool {
}
```

启动类位于 `com.aishop`，`@SpringBootApplication` 默认扫描该包及其子包。

Spring 因此创建：

```text
QueryOrderTool Bean
QueryLogisticsTool Bean
SearchProductTool Bean
SearchKnowledgeTool Bean
PrepareCancelOrderTool Bean
```

### 第二层：Registry 建立业务映射

Registry 构造器：

```java
public AssistantToolRegistry(
        List<AssistantTool> tools) {
}
```

Spring 发现参数类型是 `List<AssistantTool>`，会从容器中找到所有实现该接口的 Bean，组成 List 并传入。

概念上类似：

```java
List<AssistantTool> tools = List.of(
        queryOrderTool,
        queryLogisticsTool,
        searchProductTool,
        searchKnowledgeTool,
        prepareCancelOrderTool);

new AssistantToolRegistry(tools);
```

Registry 再遍历：

```java
for (AssistantTool tool : tools) {
    ToolPolicy policy = tool.policy();
    actions.putIfAbsent(policy.action(), tool);
    names.putIfAbsent(policy.name(), tool);
}
```

因此：

```text
Spring 负责发现并创建工具
Registry 负责建立 action/name 到工具对象的映射
```

如果工具没有 `@Component` 或 `@Bean`，虽然实现了 `AssistantTool`，也不会进入 Spring 容器和 Registry。

## 9. 构造器注入是如何发生的

代码：

```java
private final AssistantToolRegistry toolRegistry;

public NativeFunctionCallingService(
        ...,
        AssistantToolRegistry toolRegistry,
        ...) {
    this.toolRegistry = toolRegistry;
}
```

含义：

```text
构造器参数 toolRegistry
  -> Spring 从容器中传入的 Registry Bean

this.toolRegistry
  -> 当前 Service 保存的成员变量
```

Spring 4.3 以后，Bean 只有一个构造器时可以省略 `@Autowired`。

所以：

```java
this.toolRegistry = toolRegistry;
```

不是注入动作本身，只是保存已经由 Spring 注入的对象引用。

## 10. Optional Lambda 路由代码如何理解

核心代码：

```java
result = toolRegistry.find(task.action())
        .map(tool -> executeTool(
                context,
                task.taskId(),
                tool,
                task.slots()))
        .orElseGet(() -> result(
                task.taskId(),
                task.action(),
                null,
                ToolExecutionStatus.NOT_SUPPORTED,
                task.slots(),
                null,
                Map.of(),
                "当前阶段没有为该 action 注册工具"));
```

类型变化：

```text
Optional<AssistantTool>
  -> map(tool -> TaskToolResult)
  -> Optional<TaskToolResult>
  -> orElseGet 生成兜底 TaskToolResult
```

逻辑等价于：

```java
Optional<AssistantTool> optional =
        toolRegistry.find(task.action());

if (optional.isPresent()) {
    result = executeTool(
            context,
            task.taskId(),
            optional.get(),
            task.slots());
} else {
    result = buildNotSupportedResult(task);
}
```

使用 `orElseGet()` 的好处是只有工具不存在时才延迟创建兜底结果。

## 11. ToolSpecification 在哪里使用

每个 Tool 定义自己的模型工具说明：

```java
private static final ToolSpecification SPECIFICATION =
        ToolSpecification.builder()
                .name("search_products")
                .description("按关键词、分类和预算搜索在售商品")
                .parameters(JsonObjectSchema.builder()
                        .addStringProperty("query", "商品关键词")
                        .addNumberProperty("budgetMax", "最高预算")
                        .required("query")
                        .additionalProperties(false)
                        .build())
                .strict(true)
                .build();
```

它描述：

- 工具名称。
- 工具用途。
- 参数名称、类型和说明。
- 必填参数。
- 是否允许未知字段。

它不是工具实现，也不会因为定义了静态字段就自动发送给 LLM。

## 12. Registry 如何取得所有 Specification

Registry 启动时保存的是 Tool 对象。请求到达后，显式调用：

```java
public List<ToolSpecification> specifications() {
    return byName.values()
            .stream()
            .map(AssistantTool::specification)
            .toList();
}
```

等价于：

```java
for (AssistantTool tool : byName.values()) {
    result.add(tool.specification());
}
```

该方法不需要参数，是因为所有 Tool 已经保存在 Registry 的 `byName` 中。

`specifications()` 不会被 Spring 自动调用，而是在原生 Function Calling 请求中由业务代码显式调用。

## 13. ToolSpecification 如何发送给 LLM

HTTP 入口：

```http
POST /api/assistant/tools/function-call-preview
```

调用链：

```text
AssistantToolController.functionCall()
  -> NativeFunctionCallingService.preview()
  -> toolRegistry.specifications()
  -> modelGateway.propose(message, tools)
  -> LangChain4jToolCallingModelGateway.propose()
  -> ChatRequest.toolSpecifications(tools)
  -> chatModel.chat(request)
```

核心代码：

```java
ToolCallingModelReply reply =
        modelGateway.propose(
                message,
                toolRegistry.specifications());
```

进入 Gateway 后：

```java
ChatRequest request = ChatRequest.builder()
        .messages(
                SystemMessage.from(SYSTEM_PROMPT),
                UserMessage.from(message))
        .toolSpecifications(tools)
        .toolChoice(ToolChoice.AUTO)
        .temperature(0.0)
        .build();

var response = chatModel.chat(request);
```

工具不是拼接进用户文本，而是作为 `ChatRequest` 的独立结构化字段发送。

LangChain4j 会转换成 OpenAI-compatible 请求中的：

```json
{
  "messages": [...],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_products",
        "description": "按关键词、分类和预算搜索在售商品",
        "parameters": {...}
      }
    }
  ],
  "tool_choice": "auto"
}
```

模型收到用户消息和所有工具说明后，才决定是否调用工具并填写 `arguments`。

## 14. ToolSpecification 不会自动执行 Java 方法

当前使用的是底层：

```java
chatModel.chat(ChatRequest)
```

LangChain4j 负责：

```text
ToolSpecification -> API tools JSON
模型 tool_calls -> ToolExecutionRequest
```

Java 项目负责：

```text
解析 ToolExecutionRequest.arguments
  -> Registry 按 name 查找
  -> Orchestrator
  -> prepare
  -> ToolPolicy
  -> execute/PREPARED
```

所以模型只提出调用请求，工具不会被框架直接执行。

## 15. 为什么没有直接使用 LangChain4j @Tool

`@Tool` 适合把普通 Java 方法快速暴露给 LangChain4j：

```java
@Tool("查询订单")
public OrderView queryOrder(String orderNo) {
    return orderService.findByOrderNo(orderNo);
}
```

如果再通过高层 `AiServices.tools(toolObject)` 注册，LangChain4j 可以：

```text
读取 @Tool
  -> 生成 ToolSpecification
  -> LLM 选择工具
  -> 自动反射调用 Java 方法
  -> 将结果回传 LLM
```

但 `@Tool` 本身不是 Spring 注解，不会单独完成 Bean 注册、权限校验或风险判断。

## 16. 为什么当前项目选择自定义 AssistantTool

当前工具抽象：

```java
public interface AssistantTool {

    ToolPolicy policy();

    ToolSpecification specification();

    PreparedToolCall prepare(
            ToolContext context,
            Map<String, Object> arguments);

    ToolExecutionOutcome execute(
            ToolContext context,
            PreparedToolCall call);
}
```

它不仅描述工具，还表达业务执行规则。

### 16.1 显式风险策略

```text
READ_ONLY + autoExecutable=true
  -> 允许自动执行

PREPARE_ONLY + autoExecutable=false
  -> 只准备，不执行
```

LLM 无权修改 `ToolPolicy`。

### 16.2 prepare 与 execute 分离

```text
prepare
  -> 参数规范化
  -> 当前用户和资源归属检查
  -> 业务状态检查
  -> 生成操作预览

execute
  -> 真正调用业务 Service
```

高风险工具可以停在 `prepare()`，为后续 `PendingAction`、二次确认和跨请求恢复提供基础。

### 16.3 服务端用户上下文

`ToolContext` 中的用户来自 Session，不来自模型参数：

```java
orderService.findByOrderNo(
        context.user(),
        orderNo);
```

模型不能通过伪造 `userId` 查询其他用户订单。

### 16.4 工具白名单和确定性路由

```text
Planner Action -> Registry -> Tool
Function Calling name -> Registry -> Tool
```

所有模型入口共用同一个 Registry 和 Orchestrator，不会因为更换模型协议而绕过安全层。

### 16.5 结构化执行状态

统一返回：

```text
SUCCEEDED
PREPARED
NEEDS_INPUT
NOT_SUPPORTED
FAILED
SKIPPED_DEPENDENCY
```

这些状态可以直接进入后续状态机，而不是依赖分析自然语言判断执行结果。

### 16.6 更容易测试和审计

测试可以明确断言：

```java
assertThat(result.status())
        .isEqualTo(PREPARED);

assertThat(tool.executeCount)
        .hasValue(0);
```

这能证明高风险工具没有被执行，而不仅是 Prompt 中写了“不要执行”。

## 17. 自定义 Tool 的成本和边界

自定义方案不是任何场景都更优，也有明确成本：

- 需要维护 `AssistantTool`、Registry、Orchestrator 和结果模型。
- Tool Schema、Planner Action 契约和 Validator 当前存在重复定义。
- 需要自己实现工具结果回传 LLM 的 Agent Loop。
- 新增能力需要补充策略、参数校验和测试。

简单只读场景，例如计算器、天气和公开搜索，使用 `@Tool + AiServices` 通常开发更快。

当前项目选择自定义 Tool 的理由是业务包含：

```text
订单归属
取消、支付、退款等写操作
二次确认
任务依赖
跨请求恢复
审计和幂等
```

这些场景更需要显式执行边界。

## 18. 能否同时使用 @Tool 与自定义 Registry

可以，但同一条链路只能有一个真正执行者。

错误设计：

```text
AiServices 自动执行 @Tool 一次
  -> Orchestrator 又执行一次
  -> 可能产生重复操作
```

可行的混合方式：

```text
@Tool 只用于生成工具描述
  -> Registry 负责白名单
  -> Orchestrator 负责唯一执行
```

或者只对无副作用的只读工具使用 `@Tool` 自动执行，高风险工具仍然只暴露 `prepare_*` 能力。

当前项目已经手动定义 `ToolSpecification`，再增加 `@Tool` 会形成两套描述来源，因此现阶段继续使用自定义 `AssistantTool` 更清晰。

## 19. 当前设计还需要改进什么

### 19.1 统一能力目录

目前以下信息分散在：

```text
PlannerPromptFactory
PlanValidator
RuleAssistantPlanner
ToolPolicy
ToolSpecification
AssistantToolRegistry
```

后续应设计 `ActionContract` 或 `CapabilityCatalog`，统一保存：

```text
action
intent
executionMode
riskLevel
allowedSlots
requiredSlots
输入输出 Schema
ToolSpecification
是否已实现
```

Planner、Validator、Function Calling 和 Registry 都从同一能力目录读取，避免契约漂移。

### 19.2 conditions 运行时求值

当前 `dependsOn` 已经参与排序和失败传播，但 `conditions` 只完成结构校验，没有读取前置工具结果判断业务条件。

### 19.3 PendingAction

当前 `PREPARED` 只存在 HTTP 返回值中，还没有持久化为可确认、可过期、可恢复的待执行动作。

### 19.4 完整 Agent Loop

原生 Function Calling 当前只处理模型第一次工具请求，没有把工具结果作为 `ToolExecutionResultMessage` 回传给模型继续推理和生成最终回答。

## 20. 本阶段最关键的理解

1. 第二阶段不是重复第一阶段，而是把 Plan 转换成受控业务调用。
2. Planner 主链路不需要在 Plan 生成后再调用一次 LLM 选择工具。
3. 原生 Function Calling 是替代入口和学习支线。
4. LLM 通过 Planner Prompt 知道 Action，通过 ToolSpecification 知道原生工具名。
5. Registry 不是自己扫描类，而是依赖 Spring 的 Bean 扫描和 `List<AssistantTool>` 集合注入。
6. ToolSpecification 会显式放入 `ChatRequest.toolSpecifications(tools)`，不会自动发送。
7. 模型只提出调用请求，Java Registry、Policy 和业务 Service 掌握执行权。
8. 自定义 Tool 的核心价值不是少写代码，而是显式表达业务安全和状态边界。

## 21. 面试亮点：为什么使用自定义 Tool 而不是直接使用 @Tool

面试中不要只说“我没有使用 `@Tool`”，而要说明做过技术取舍。

推荐表达：

> 我评估过 LangChain4j 的 `@Tool + AiServices` 自动调用方式，它适合计算器、天气、公开检索等简单只读工具，但电商客服包含订单查询、取消、退款、支付和改地址等带用户资源归属与业务副作用的操作。为了避免模型选中工具后立即执行，我设计了自定义 `AssistantTool` 抽象，将 `ToolSpecification`、`ToolPolicy`、`prepare()` 和 `execute()` 分离。模型只负责提出工具请求，Java Registry 负责白名单路由，ToolContext 绑定当前登录用户，ToolPolicy 只允许 READ_ONLY 自动执行，高风险工具只返回 PREPARED。这样可以自然扩展 PendingAction 二次确认、过期控制、重新鉴权和幂等执行。

可以重点展开以下亮点。

### 亮点一：不是依赖 Prompt 保证安全

```text
Prompt 约束模型
  -> Validator 校验计划
  -> Registry 限制工具集合
  -> ToolPolicy 限制执行方式
  -> ToolContext 限制用户范围
  -> Service 校验资源归属和业务状态
```

即使模型判断错误，也不能直接获得业务写权限。

### 亮点二：模型选择与业务授权解耦

```text
LLM：建议做什么
Java：是否支持、是否允许、如何执行
```

Planner 和原生 Function Calling 可以更换，但后端安全执行层保持不变。

### 亮点三：prepare/execute 为状态恢复打基础

```text
低风险：prepare -> execute
高风险：prepare -> PendingAction -> 用户确认
       -> 重新鉴权和状态复检 -> execute
```

这是从普通 Tool Calling 演进到可恢复业务 Agent 的关键设计。

### 亮点四：结构化结果支持状态机

```text
SUCCEEDED
PREPARED
NEEDS_INPUT
FAILED
SKIPPED_DEPENDENCY
```

上层不需要从自然语言中猜测工具结果，可以确定性决定下一步。

### 亮点五：一套安全层复用两种模型协议

```text
Planner Action 路线
原生 ToolExecutionRequest 路线
  -> 共用 Registry、Orchestrator 和 ToolPolicy
```

这说明系统没有把安全能力绑定在某个 LLM SDK 上。

## 22. 面试时需要主动说明的不足

为了让表达可信，应同时说明当前边界：

- `conditions` 还没有运行时求值。
- `PREPARED` 还没有持久化为 `PendingAction`。
- 当前原生 Function Calling 不是完整多轮 Agent Loop。
- Planner 契约与 Tool 元数据需要统一到 `CapabilityCatalog`。
- 当前高风险 Tool 只准备，不会真正修改订单。

能够主动说明边界，比声称“实现了全自动电商 Agent”更有技术含量。

## 23. 第二阶段一句话总结

> 第二阶段实现了 Planner Action 和原生 Function Calling 到 Java 业务工具的统一安全执行层；项目选择自定义 `AssistantTool` 而不是直接依赖 `@Tool` 自动执行，是为了显式控制工具白名单、当前用户、风险等级、准备与执行分离及结构化状态，为后续 PendingAction 二次确认和跨请求恢复提供基础。
