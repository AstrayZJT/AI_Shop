# 第七阶段评测与可观测性源码学习文档

本文对应当前 Goal 的第七阶段，对应实施计划书中的阶段八。

## 1. 阶段目标

前面的阶段已经完成 Planner、Tool Calling、RAG、上下文、状态机和 Guardrails，但页面能回答不能证明 Agent 质量稳定。本阶段解决两个问题：

1. 用带人工标签的离线数据回答 Planner 做得准不准。
2. 用请求级 trace 回答一次请求为什么选择这个任务和工具。

评测是批量、可重复的实验；trace 是单次请求的结构化执行摘要，不能互相替代。

## 4. 评测集格式

每一行 JSONL 是一个独立案例，包含 id、message、expectedPlanType、expectedIntents、expectedActions、expectedTaskCount、expectedSlotsByAction 和 expectedDependsOnByAction。

多任务案例的核心标签是：

    expectedActions = [QUERY_LOGISTICS, CANCEL_ORDER]
    expectedDependsOnByAction.CANCEL_ORDER = [t1]

只比较 intent 不够，因为 ORDER 可能是查询，也可能是取消。只比较任务数量也不够，因为没有 dependsOn 的取消任务可能绕过物流检查。

slots 采用关键字段标注。订单号、预算等确定性字段精确比较；自然语言 query 不强制比较完整句子，避免把合理的同义表达误判为失败。

## 5. 两种评测模式

PlannerEvaluationMode 有 RULE_BASELINE 和 ACTIVE_PLANNER。

RULE_BASELINE 直接调用 RuleAssistantPlanner，不访问网络、不需要 API Key，适合 CI、规则回归和验证评测器。结果的 Planner source 固定记录为 RULE_FALLBACK。

ACTIVE_PLANNER 调用 PlannerFacade，完整经过 AI 开关、Key 检查、LLM JSON 解析、PlanValidator、PlanSemanticGuard 和规则 fallback。显式运行该模式可能产生 57 次模型请求，不能作为普通单元测试默认执行。

接口：

    GET /api/assistant/evaluation/planner?mode=RULE_BASELINE

Controller 先执行 AuthService.requireUser，因此不是匿名公网评测接口。

## 6. 指标计算

PlannerEvaluationService.evaluate 逐条生成计划，并分别比较：

### intentAccuracy

实际任务的 intent 列表必须和标注列表相同。

    intentAccuracy = 命中案例数 / 总案例数

### actionExactMatch

比较 action 的数量、顺序和名称。缺少任务、顺序变化或动作错误都会失败。

### slotExactMatch

按 action 找实际任务，再比较标注的关键槽位。数字统一转成字符串，避免 JSON 的 Integer 和 Long 类型差异污染指标。

### taskCountAccuracy

实际任务数量等于期望数量才命中。

### multiTaskAccuracy

判断实际和标注是否都属于多任务：

    actual.tasks.size() > 1 == expectedTaskCount > 1

### dependencyExactMatch

按 action 找到任务，比较其 dependsOn 列表。它能发现任务数量正确但执行依赖错误的问题。

### ruleFallbackRate

统计 PlannerResult.source 为 RULE_FALLBACK 的比例。该指标必须结合 action 准确率和失败原因分析，不能单独判断好坏。

## 7. 测试隔离真实模型

PlannerEvaluationServiceTest 使用真实 RuleAssistantPlanner，mock PlannerFacade，只运行 RULE_BASELINE。因此 CI 不访问远程模型。

已有测试的职责分别是：

    PlannerEvaluationServiceTest：评测算法、数据集和规则基线
    LlmAssistantPlannerTest：固定模型响应的 JSON 解析
    PlannerFacadeTest：API Key 缺失、模型失败和非法计划 fallback

这比把所有行为塞进一个依赖网络的集成测试更容易定位问题。

## 9. Trace 如何进入正式链路

调用链：

    AssistantController.chat
    -> AssistantService.chat
    -> AssistantAgentService.handle
    -> Context
    -> Planner 或 StateMachine.resume
    -> Tool/RAG
    -> AnswerComposer
    -> AgentTrace
    -> ChatResponse.trace

非恢复请求中，planningLatencyMs 统计 PlannerFacade.plan 和 ConversationPlanResolver，executionLatencyMs 统计 AssistantStateMachine.start。

恢复请求不会重新调用 Planner，因此 planningLatencyMs 为 0，executionLatencyMs 统计 AssistantStateMachine.resume。

answerLatencyMs 统计 RAG AnswerComposer 和 AssistantAnswerComposer。totalLatencyMs 从 handle 开始计算，还包括等待状态查询和 Context 构建，所以各阶段相加可能略小于总耗时。

## 10. 为什么不记录完整 Prompt 和 arguments

工具参数可能包含订单号、地址和用户长文本。trace 的作用是解释任务、工具、状态和耗时，不需要复制全部业务数据。

当前只保留 targetRef 和截断后的 message，不保留完整 arguments Map、原始 Prompt 和完整模型输出。日志包含 traceId、Planner 来源、任务数、PlanRun 状态、耗时和 Token，之后可以用 planRunId 查询 PlanRun、TaskRun、PendingAction 和审计表。

## 11. 四个面试演示

### 商品推荐

输入：推荐一款 500 元以内的耳机。

解释 Planner 生成 SEARCH_PRODUCT，Tool 查询商品，trace 展示 action、toolName、status 和耗时。

### 商品与 RAG 组合

输入：推荐一款耳机，并说明不合适能否七天无理由退货。

解释 Planner 拆出 SEARCH_PRODUCT 和 SEARCH_KNOWLEDGE，RAG 返回 citation，AnswerComposer 合成回答，trace 展示 ragMode、sourceCount 和 answerMode。

### 高风险确认恢复

输入：查订单物流，如果没发货就取消。

解释 t1 是 QUERY_LOGISTICS，t2 是 CANCEL_ORDER 且 dependsOn=[t1]。条件满足后创建 PendingAction，请求结束；确认后重新校验并恢复 PlanRun。

### 模型失败与越权

关闭真实模型时，trace 显示 RULE_FALLBACK 和 AI_DISABLED。越权确认时返回 403，PendingAction 和订单状态不变化。

## 12. 正确解释基线 1.0

基线 1.0 证明：

- 数据集可解析。
- 当前规则在当前样本上符合标签。
- 指标计算可以重复。
- 规则改动会触发回归失败。

它不能证明真实模型泛化能力、所有用户表达都能识别，或者线上业务天然安全。真实模型评测还应记录模型版本、Prompt 版本、失败样本和输入分布，并与调 Prompt 的样本分离。

## 13. 当前边界

本阶段是学习和面试级可观测性，不是完整生产 tracing 平台。当前没有 OpenTelemetry、Prometheus、Jaeger、跨服务 trace propagation、评测结果历史表和大规模对抗数据集。

简历中可以写：

    实现结构化 Planner 离线评测和请求级 Agent trace

不应写成完整企业级可观测平台。

## 14. 面试表达

可以这样回答：

> 我没有只靠手工聊天判断 Agent 质量，而是建立了 57 条 JSONL 结构化评测集，分别标注 intent、action、关键 slots、任务数量和 dependsOn。评测器支持规则基线和显式的真实 Planner 模式，统计 intent accuracy、action exact match、slot match、多任务准确率、依赖准确率和规则兜底率。线上请求通过 AgentTrace 记录 traceId、Planner 来源、模型 Token、Planner/状态机/回答耗时、任务工具结果和 RAG 模式。这样可以区分是模型规划错误、后端校验拒绝、工具执行失败还是回答合成问题。

如果被问为什么基线 1.0 不能说明模型 100%，回答：

> 规则基线和真实模型是两个被测对象。基线 1.0 只说明当前确定性规则在当前标注集上通过；真实模型还会受到表达泛化、模型版本、Prompt 版本和输入分布变化影响。真实模型需要显式运行 ACTIVE_PLANNER，并保存每条失败样本和版本信息。

## 15. 阶段结论

1. Agent 必须有离线评测，不能只靠页面演示。
2. intent 正确不代表 action、slots 和依赖正确。
3. 规则兜底率必须和准确率、失败原因一起分析。
4. trace 用于解释单次请求，不能替代带标签评测。
5. 结构化 trace 比自然语言日志更容易查询。
6. 可观测性不能复制完整 Prompt 和敏感业务参数。
7. 简历指标必须能由固定数据集或运行数据复现。
8. 当前实现适合学习和面试，生产环境还需要完整 tracing、指标系统和更大评测集。
