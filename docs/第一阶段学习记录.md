# 第一阶段学习记录

> 学习日期：2026-07-15
> 学习主题：LLM Planner、结构化计划、计划校验、意图边界和任务依赖

## 1. 第一阶段到底完成了什么

第一阶段解决的核心问题是：

> 把用户自然语言转换成一份经过 Java 后端校验的结构化任务计划。

完整链路：

```text
用户输入
  -> AssistantPlannerController
  -> PlannerFacade
  -> LlmAssistantPlanner
  -> PlannerPromptFactory
  -> LangChain4jPlannerModelGateway
  -> chatModel.chat(request)
  -> 模型返回 JSON 文本
  -> Jackson 反序列化为 AssistantPlan
  -> PlanValidator 结构和业务契约校验
  -> PlanSemanticGuard 意图边界校验
  -> PlannerResult
```

第一阶段的结果只是计划，不会查询订单、调用业务工具或者修改数据库。工具选择和受控执行属于第二阶段。

更准确地说，第一阶段实现的是“任务规划和编排描述”，不是“任务运行时编排”。

## 2. PlannerFacade 的作用

`PlannerFacade` 是 Planner 模块的统一入口，负责串联和控制整个规划流程：

1. 校验用户输入长度和上下文数量。
2. 判断 AI 是否启用、API Key 是否存在。
3. 调用 LLM Planner。
4. 校验模型生成的计划。
5. 检查用户原话与计划之间的意图边界。
6. 模型失败时切换到本地规则 Planner。
7. 统一返回 `PlannerResult`。

核心代码：

```java
LlmAssistantPlanner.LlmPlannerOutput output = llmPlanner.plan(input);
AssistantPlan validated = planValidator.validate(output.plan());
semanticGuard.validate(input, validated);
```

因此，`PlannerFacade` 不直接理解用户意图，也不直接调用模型。它是规划流程的协调者和降级入口。

相关源码：

- `src/main/java/com/aishop/assistant/planner/PlannerFacade.java`
- `src/main/java/com/aishop/assistant/planner/LlmAssistantPlanner.java`
- `src/main/java/com/aishop/assistant/planner/RuleAssistantPlanner.java`

## 3. 真正调用 LLM 的位置

真正发起模型请求的位置是：

```java
var response = chatModel.chat(request);
```

位于：

```text
src/main/java/com/aishop/assistant/planner/LangChain4jPlannerModelGateway.java
```

`ChatRequest` 主要规定：

- System Prompt：模型身份、可用枚举、任务契约和 JSON 示例。
- User Prompt：用户消息、会话摘要和最近消息。
- `temperature`：输出随机性。
- `maxOutputTokens`：最大输出长度。
- `ResponseFormat.JSON`：要求模型输出 JSON。

它不会自动保证：

- 字段一定完整。
- `action` 与 `executionMode` 一定匹配。
- 订单号一定真实。
- 任务依赖一定合法。
- 用户只咨询规则时模型不会错误规划写操作。

因此，请求模型输出 JSON 只是第一层约束，不能代替 Java 校验。

## 4. 为什么 `reply.text()` 得到的是 JSON 字符串

这里存在两层数据：

```text
DashScope HTTP 响应
  -> LangChain4j 解析外层响应
  -> 提取 assistant message.content
  -> reply.text() 返回 content 字符串
```

由于 Prompt 和 `ResponseFormat.JSON` 要求模型把 `message.content` 写成 JSON，所以：

```java
String rawOutput = reply.text();
```

得到的通常是类似下面的字符串：

```json
{"planType":"SINGLE_TASK","tasks":[...],"summary":"查询订单"}
```

但它此时仍然只是 `String`，还不是 Java 对象，也不能直接执行。

## 5. 严格 Jackson 如何发挥作用

构造器中创建了一个可重复使用的严格 `ObjectReader`：

```java
ObjectMapper strictMapper = objectMapper.copy()
        .enable(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES)
        .enable(DeserializationFeature.FAIL_ON_TRAILING_TOKENS)
        .enable(JsonParser.Feature.STRICT_DUPLICATE_DETECTION);

this.planReader = strictMapper.readerFor(AssistantPlan.class);
```

构造器阶段只是配置解析器，没有解析任何模型输出。真正生效的位置是：

```java
AssistantPlan plan = planReader.readValue(rawOutput);
```

三项严格配置分别防止：

- `FAIL_ON_UNKNOWN_PROPERTIES`：模型发明未知字段。
- `FAIL_ON_TRAILING_TOKENS`：合法 JSON 后继续夹带其他内容。
- `STRICT_DUPLICATE_DETECTION`：同一个字段重复出现并覆盖前值。

Jackson 会根据 Java Record 的字段名和类型，把 JSON 创建为：

```text
AssistantPlan
  -> List<AssistantTask>
```

例如字符串枚举 `"CANCEL_ORDER"` 会转换为 `AssistantAction.CANCEL_ORDER`。不存在的枚举会解析失败。

## 6. 为什么解析后还需要 PlanValidator

Jackson 主要检查“能否转换成 Java 类型”，`PlanValidator` 检查“这份计划在业务上是否成立”。

主要校验包括：

- `SINGLE_TASK` 必须只有一个任务。
- `MULTI_TASK` 至少包含两个任务。
- 一次最多生成 5 个任务。
- `taskId` 不能为空、不能重复。
- `dependsOn` 引用的任务必须存在。
- 任务不能依赖自己，依赖图不能有环。
- `action`、`intent`、`executionMode` 必须匹配。
- `slots` 只能包含当前动作允许的参数。
- 必填参数必须存在，或者声明在 `missingSlots` 中。
- `conditions` 的字段、运算符和值必须符合契约。
- `confidence` 如果存在，必须位于 `0~1`。

因此校验分为三层：

```text
JSON 语法和 Java 类型
  -> Jackson

字段组合和业务契约
  -> PlanValidator

用户原话与计划的意图边界
  -> PlanSemanticGuard
```

## 7. PlanSemanticGuard 做了什么

`PlanSemanticGuard` 会对照用户原话与模型计划，重点防止：

```text
用户只是咨询规则
  -> LLM 却规划取消、支付、退款、改地址等写动作
```

例如：

```text
用户：怎么取消订单？
错误计划：CANCEL_ORDER + ASK_CONFIRM
正确计划：SEARCH_KNOWLEDGE + TOOL_READ
```

当前实现还是最小关键词规则，主要判断：

```text
consultation && !explicitExecution && containsWriteAction
```

它不是完整的自然语言安全分类器，因此可能：

- 误拒绝：用户用了系统不熟悉的表达方式。
- 误放行：复杂语句同时包含咨询和执行意图。

正确的后续方向不是无限增加关键词，而是引入结构化 `requestType`、原文证据、`GuardDecision` 和高风险 `PendingAction` 二次确认。

## 8. 计划字段由谁填写

正常情况下，具体字段值由 LLM 生成，但 Java 后端定义字段结构、允许取值和安全规则。

```text
字段结构          -> Java Record 定义
允许取值          -> Java enum 定义
具体计划值        -> LLM 生成
JSON 转 Java      -> Jackson 完成
字段是否合法      -> PlanValidator 判断
意图是否越界      -> PlanSemanticGuard 判断
模型失败后的字段  -> RuleAssistantPlanner 生成
最终执行权限      -> 后端 ToolPolicy 决定
```

最重要的原则是：

> LLM 拥有理解权和规划建议权，但没有权限定义权和最终执行权。

## 9. AssistantPlan 与 AssistantTask 的区别

`AssistantPlan` 是整份计划：

```java
public record AssistantPlan(
        PlanType planType,
        List<AssistantTask> tasks,
        String summary
) {}
```

`AssistantTask` 是计划中的单个任务：

```java
public record AssistantTask(
        String taskId,
        AssistantIntent intent,
        AssistantAction action,
        ExecutionMode executionMode,
        Map<String, Object> slots,
        List<String> missingSlots,
        List<String> dependsOn,
        List<TaskCondition> conditions,
        Double confidence,
        String reason
) {}
```

无论单任务还是多任务，`tasks` 中的每个元素都使用同一套 `AssistantTask` 字段。讲解时省略字段的 JSON 只是示意，不能作为当前项目的真实模型输出。

## 10. 每个字段后续有什么用途

| 字段 | 作用 |
|---|---|
| `planType` | 描述单任务、多任务、追问或不支持 |
| `taskId` | 唯一标识任务，供依赖和结果映射使用 |
| `intent` | 业务领域分类，例如订单、商品、售后 |
| `action` | 具体业务动作，后续作为工具路由键 |
| `executionMode` | 声明回答、只读、草稿、确认或追问模式 |
| `slots` | 从用户原话提取的工具参数 |
| `missingSlots` | 缺失的必填参数，后续用于追问 |
| `dependsOn` | 定义任务先后顺序和失败传播 |
| `conditions` | 描述前置结果满足什么条件时继续 |
| `confidence` | 模型自评置信度，不是可靠概率 |
| `reason` | 解释单个任务为什么这样规划 |
| `summary` | 整份计划摘要 |

其中：

- `action` 后续用于找到具体工具。
- `slots` 后续作为 `tool.prepare()` 的参数。
- `missingSlots` 不为空时应该暂停并追问。
- `executionMode` 是计划声明，真正权限仍由 `ToolPolicy` 控制。
- `reason` 和 `summary` 用于解释、日志和调试，不能控制业务执行。

## 11. Action 与 ExecutionMode 的安全契约

后端固定了动作与执行模式的映射，例如：

```text
QUERY_ORDER       -> TOOL_READ
QUERY_LOGISTICS   -> TOOL_READ
SEARCH_PRODUCT    -> TOOL_READ
SEARCH_KNOWLEDGE  -> TOOL_READ
CREATE_ORDER_DRAFT -> CREATE_DRAFT
CANCEL_ORDER      -> ASK_CONFIRM
PAY_ORDER         -> ASK_CONFIRM
REQUEST_REFUND    -> ASK_CONFIRM
CONFIRM_RECEIPT   -> ASK_CONFIRM
UPDATE_ADDRESS    -> ASK_CONFIRM
ASK_CLARIFICATION -> CLARIFY
GENERAL_CHAT      -> ANSWER_ONLY
```

即使 LLM 返回：

```text
CANCEL_ORDER + TOOL_READ
```

也会被 `PlanValidator` 拒绝。

这能防止模型自行把高风险动作降级成可自动执行动作。不过第一阶段仍然没有真正执行取消操作。

## 12. dependsOn 如何发挥作用

`dependsOn` 保存当前任务直接依赖的前置 `taskId`：

```json
{
  "taskId": "t2",
  "action": "CANCEL_ORDER",
  "dependsOn": ["t1"]
}
```

含义是：`t2` 必须等待 `t1` 完成。

第一阶段负责生成和校验这个依赖描述：

- 被依赖的任务必须存在。
- 任务不能依赖自身。
- 任务依赖不能形成环。

第二阶段的 `AssistantToolOrchestrator` 才真正使用它：

1. 根据 `dependsOn` 计算任务入度。
2. 使用拓扑排序确定执行顺序。
3. 执行任务前检查依赖任务结果。
4. 依赖失败时将后续任务标记为 `SKIPPED_DEPENDENCY`。

必须区分：

```text
dependsOn  -> 谁先执行、前面失败是否继续
conditions -> 前置结果满足什么业务条件才继续
```

当前项目的 `conditions` 只完成格式校验，还没有在 Orchestrator 中真正求值。`dependsOn` 也不会自动把前置任务输出绑定到后续任务参数。

## 13. confidence 应该怎么理解

`confidence` 是 LLM 自己填写的主观评分，例如：

```json
"confidence": 0.95
```

它不是经过统计校准的真实正确率。当前代码只检查它是否在 `0~1`，还没有根据阈值做决策。

即使高风险动作的置信度是 `0.99`，也不能直接执行。后续仍然必须完成：

- 当前用户鉴权。
- 目标资源归属检查。
- 业务状态检查。
- 创建待确认动作。
- 用户二次确认。
- 确认后重新鉴权和状态复检。
- 幂等执行。

## 14. LLM 失败时如何处理

以下情况会触发本地规则 Planner：

- AI 未开启。
- API Key 缺失。
- 模型调用失败或超时。
- 模型返回空内容。
- JSON 无法解析。
- 计划违反业务契约。
- 计划违反当前语义边界规则。

本地 `RuleAssistantPlanner` 直接使用 Java 构造 `AssistantPlan`，生成后仍然经过相同的 `PlanValidator` 和 `PlanSemanticGuard`。

因此降级不是绕过安全校验，而是更换计划来源。

## 15. 第一阶段当前边界

第一阶段已经完成：

- LLM 结构化意图识别和任务规划。
- 多任务拆分与依赖描述。
- 参数抽取与缺失槽位声明。
- 严格 JSON 解析。
- 计划结构和业务契约校验。
- 最小咨询与执行边界保护。
- 本地规则降级。

第一阶段尚未完成：

- 业务工具调用。
- `conditions` 运行时求值。
- 通用多任务冲突检测。
- 前置任务输出到后续任务的参数绑定。
- 高风险动作持久化确认。
- 跨请求暂停和恢复。
- 确认后的重新鉴权、状态复检和幂等执行。

因此，不能把第一阶段描述成“已经实现自动取消订单”，应该描述成：

> 已实现基于 LLM 的结构化任务规划，并通过 Java 契约校验、语义边界检查和规则降级，将不稳定的模型输出转换成可由后续编排器安全消费的任务计划。

```text
PlannerResult
  -> AssistantToolOrchestrator
```

> Planner 负责提出“做什么”，Orchestrator 负责安排“怎么做”，ToolPolicy 和业务服务负责决定“是否允许做”。
